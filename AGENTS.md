# AGENTS.md

本文件用于指导后续开发本项目的代码。项目目标是基于 `Data` 目录中的手表运动数据，完成解析、清洗、SQL Server 入库、查询分析和前端/移动端展示。后续实现时应优先保证数据结构清晰、入库可重复、分析结果可校验。

## 1. 项目数据约定

`Data` 目录当前包含 4 次运动：

- `run1`：跑步
- `run2`：跑步
- `ride1`：骑行
- `strength1`：力量训练

每次活动可能包含以下文件：

- `.fit`：二进制原始运动文件，暂时作为原始文件归档；如果后续引入 FIT 解析库，再作为补充数据源。
- `.gpx`：轨迹点数据，优先用于解析经纬度、海拔、时间、心率和步频。
- `.tcx`：训练结构数据，优先用于解析活动、分段、圈、距离、时长、速度、心率和功率。
- `json *.txt`：活动摘要 JSON，优先用于解析活动基础信息和平台字段。
- `纯文本报告 *.txt`：人工可读报告，主要用于核对解析结果，不建议作为核心结构化数据源。

实现代码时不要假设所有运动类型都有完整轨迹或完整功率数据。力量训练可能缺少轨迹和分段数据；骑行和跑步字段也可能不完全一致。

## 2. 推荐代码结构

后续新增代码时，建议按职责拆分，避免把解析、入库和展示逻辑写在同一个文件中。

建议结构：

```text
src/
  parsers/
    json_parser.*
    gpx_parser.*
    tcx_parser.*
    fit_parser.*        # 可选，后续扩展
  db/
    schema.sql
    seed_or_import.*
    queries.sql
  services/
    import_service.*
    analysis_service.*
  api/
    activities.*
  app/ or web/
    ...
```

开发顺序建议：

1. 先实现 JSON 解析，得到活动主表需要的摘要字段。
2. 再实现 TCX 解析，得到活动分段、圈和训练指标。
3. 再实现 GPX 解析，得到轨迹点。
4. 最后再考虑 FIT 解析、App 接口和可视化页面。

## 3. 数据模型与字段设计

数据库使用 SQL Server。表结构应围绕“一次活动”建立主外键关系。

建议至少包含以下表：

- `Activities`：活动主表，一次运动一条记录。
- `TrackPoints`：轨迹点表，一次采样一条记录。
- `Laps`：分段/圈表，一圈或一公里一条记录。
- `Metrics`：统计指标表，用于保存心率区间、功率区间、训练负荷等非固定字段。
- `SourceFiles`：原始文件表，用于记录每次活动关联了哪些 FIT、GPX、TCX、JSON、TXT 文件。

字段设计原则：

- 主键使用数据库内部 ID，外部平台活动 ID 单独保存。
- 时间字段同时保留 UTC 时间和本地时间时，应在字段名中明确区分。
- 数值字段优先保存原始单位或标准单位，不要只保存格式化字符串。
- 距离建议保存为米，时长保存为秒，速度保存为 m/s，心率保存为 bpm，功率保存为 W。
- 配速可以通过 SQL 或服务层计算，不建议只存储 `4:32 /km` 这种文本。
- 可空字段必须明确处理，不要用 `0` 代替未知值。

## 4. 解析实现要求

解析器应输出统一的中间数据结构，再由导入服务负责写入数据库。

推荐中间结构：

- `activity`：活动摘要。
- `trackPoints`：轨迹点数组。
- `laps`：分段数组。
- `metrics`：统计指标数组。
- `sourceFiles`：原始文件列表。

解析规则：

- JSON 是活动摘要的优先来源。
- TCX 是分段和训练结构的优先来源。
- GPX 是轨迹点的优先来源。
- 纯文本报告只用于补充或校验，不作为第一优先级解析来源。
- 同一次活动的文件应通过文件名中的 `run1`、`run2`、`ride1`、`strength1` 或活动开始时间进行关联。
- 导入过程应可重复执行。重复导入同一活动时，应更新已有记录或先清理该活动相关数据，避免产生重复轨迹点。

编码和中文处理：

- 所有新增源码和 Markdown 文件使用 UTF-8。
- 读取 `Data` 中的中文文件名时，不要硬编码非 UTF-8 字节。
- 如果报告内容出现乱码，应优先检查源文件编码，不要在业务逻辑中写死乱码替换。

## 5. SQL Server 开发要求

SQL 脚本应可重复运行或明确标注运行顺序。

建议拆分：

- `schema.sql`：建表、主键、外键、索引。
- `import.sql` 或导入程序：插入解析后的数据。
- `queries.sql`：作业展示用分析查询。

必要索引：

- `Activities(activity_type, start_time)`
- `TrackPoints(activity_id, sample_time)`
- `Laps(activity_id, lap_index)`
- `Metrics(activity_id, metric_type)`

常用查询至少覆盖：

- 活动列表和活动详情。
- 不同运动类型的总距离、总时长、平均心率。
- 单次跑步的分段配速和平均心率。
- 单次活动的轨迹点序列。
- 心率区间或功率区间分布。

## 6. 后端/API 编写建议

如果实现后端服务，接口应围绕活动查询和文件导入设计。

建议接口：

- `POST /activities/import`：导入 `Data` 中的一组活动文件或上传文件。
- `GET /activities`：返回活动列表。
- `GET /activities/{id}`：返回活动摘要。
- `GET /activities/{id}/track-points`：返回轨迹点。
- `GET /activities/{id}/laps`：返回分段数据。
- `GET /activities/{id}/metrics`：返回统计指标。

接口返回值应使用结构化 JSON，避免返回已经拼接好的展示文本。前端或 App 负责格式化配速、时间和单位。

## 7. 分析与展示重点

优先实现能够体现数据库分析价值的功能：

- 跑步：距离、用时、平均配速、分段配速、平均心率、最大心率、功率、训练负荷。
- 骑行：距离、用时、速度、心率、海拔、轨迹。
- 力量训练：训练时长、心率、热量、训练负荷。
- 对比分析：不同运动类型在距离、时长、热量、心率上的差异。

展示层不要直接解析原始文件，应只调用后端接口或读取数据库查询结果。

## 8. 验证要求

每新增一个解析器或导入流程，都应至少验证以下内容：

- 能识别 `Data` 目录中的 4 次活动。
- JSON、TCX、GPX 文件能按活动正确归组。
- 跑步活动能解析出活动摘要、分段和轨迹点。
- 骑行活动能解析出活动摘要和轨迹点。
- 力量训练活动即使缺少完整轨迹，也不会导致导入失败。
- SQL Server 中不会因为重复导入产生重复活动或重复轨迹点。
- 查询结果与 `纯文本报告 *.txt` 中的距离、时长、心率等摘要值大体一致。

## 9. 开发注意事项

- 不要把 `Data` 目录中的原始文件改名、移动或覆盖。
- 不要把数据库连接字符串、账号、密码写死在源码中。
- 不要把纯文本报告当作唯一数据源。
- 不要在数据库中只存格式化文本，应保存可计算的数值。
- 不要让前端或 App 直接读取本地原始文件；应通过后端或数据库查询获取结构化结果。
- 新增代码时优先保持实现简单清楚，先完成可运行的导入和查询，再扩展复杂可视化。
